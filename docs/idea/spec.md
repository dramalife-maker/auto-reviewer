# 1on1 Reviewer 雲端 Web 服務 — 設計規格

> 狀態：**設計階段**（尚未進 MVP）
> 用途：作為後續 MVP 實作的依據，可直接餵給 Claude Code 當實作藍本。
> 技術棧：**雲端集中式 web 服務**——Rust 後端（常駐服務）部署於雲端主機、瀏覽器前端；同機 clone 各專案 repo、預登入 `glab`/`claude` CLI；供多位主管共用。
> 修訂：本版為雲端 web 服務（非桌面 app）；雙軌架構（1on1 週報 + MR review）；skill／後端邊界；**趨勢頁資料來源已定：讀既有 1on1 檔案**（`index.md` / `YYYY-MM.md`）。

---

## 0. 背景與目標

### 問題
團隊有多個專案，每位工程師可能跨多個專案。現有的 reviewer skill 在 Claude Code 手動執行，產出 1on1 用的觀察報告。痛點：

- 要逐專案手動跑，專案一多很麻煩。
- 跑完後還要自己去資料夾翻 md，找出哪些有變更。
- 缺乏跨月份的長期觀察視角。

### 目標
一個**雲端 web 服務**，作為 reviewer skill 的 **排程器 + 控制台 + 報告閱讀器**，涵蓋**兩條軌道**：

1. **1on1 週報軌道**：定時（每週）自動批次跑所有專案的 reviewer skill，掃 commit 產出以人為中心的觀察報告。
2. **MR review 軌道**：高頻輪詢（預設每小時）偵測各專案有無待 review 的 MR，自動 review 並產出**草稿**，由管理者在 web 介面看過後手動發佈。
3. 跑完主動通知，且能直接在 web 介面看重點報告 / MR 草稿，不必 SSH 翻伺服器檔案。
4. 以「人」為中心檢視，支援跨專案彙整與跨月份長期觀察。
5. 支援單一專案獨立立即執行，不必等整批。
6. **兩軌打通**：MR review 產生的工程師觀察，會沉澱進該人的 1on1 週報（見 §6.5）。

### 部署形態
- **後端**：Rust 常駐程序，部署於雲端主機（systemd / container），內建排程與 worker pool。
- **前端**：瀏覽器 SPA，多位主管共用同一後端（認證機制 MVP 後再定，見 §8）。
- **執行環境**：後端同機維護各專案 git clone 目錄；`glab` / `claude` CLI 預先於伺服器登入，skill 子行程在此環境執行。
- **資料**：設定與操作狀態存 SQLite（`reviewer.db`）；報告與**趨勢長期資料**以檔案為主。根目錄由環境變數 **`REVIEWER_DATA_DIR`** 指定（見 §9、schema §0）。

### 非目標（本階段明確排除）
- **不接 Telegram bot 雙向問答**：AI 主動向工程師發問澄清的比例低，投報率不高，且非同步狀態管理成本高。改以「待確認項目」標記，由管理者在 1on1 當面問。
- **不做數量型指標**（commit/MR 數的長條圖等）：量多寡不代表價值，避免製造假訊號。
- **不主動點出跨專案活躍度落差**：專案進入維護期、無新功能很正常，中性陳述即可。
- **MR review 一律不自動發佈**：AI 產草稿，發佈到 GitLab（`glab mr note`）或 merge 一律需管理者在 web 介面人工確認。無人值守下絕不寫入 GitLab。

> 註（原非目標「不做 GitLab API 整合」已移除）：MR review 軌道必須讀 MR，透過**伺服器上已登入的 `glab` CLI** 取得，後端不管 token / Project ID（由 CLI 設定檔管理）；讀取為自動，**發佈為人工**。

## 1. 資訊架構

Web 介面分為六個頁面。核心是「報告閱讀器」，MR review 收件匣為第二軌的入口，其餘為控制與設定。

```
┌─ 報告閱讀器（核心）  選人 → 專案 Tab → 本週 / 趨勢
├─ MR review 收件匣    第二軌：待發佈草稿清單，看 → 編輯 → 發佈 / 忽略
├─ 控制台首頁          總覽統計、最近報告、排程狀態、全部執行
├─ 專案設定            名稱、`repo_path`、`git_remote_url`（+ 清單列上的立即執行）
├─ 人員設定            工程師 + 多個 git author identity
└─ 排程               週期/時間、逾時、MR 輪詢間隔、執行紀錄
```

歸戶（attribution）的核心原則：**以人為中心，一次設定到位**。
工程師的 git author identity 集中在「人員設定」管理，專案不需逐一設定歸戶；專案的「參與工程師」由 identity 自動比對得出（唯讀）。

## 2. 報告閱讀器（核心頁面）

### 2.1 階層結構

```
左欄：工程師清單
  └ 頂部 Tab：總覽 / 各專案（以專案名為 Tab 名）
      └ 次層 Tab：本週 / 趨勢
```

選一個人 → 預設落在「總覽 / 本週」。切專案 Tab 看單一專案的完整內容，切「趨勢」看跨月觀察。

### 2.2 左欄：工程師清單（以人為單位）

- 列出所有工程師，每人顯示參與專案數（例「2 專案」）。
- 狀態標記掛在「人」身上：
  - 未讀：藍點。
  - 有待確認項目：⚠️ 警示圖示。
  - 全部已讀：灰階 / 「已讀」字樣。

### 2.3 「總覽」Tab（跨專案彙整）

由上而下：

1. **一句話摘要**：AI 將此人本週整體狀態濃縮成 2–3 句。開 1on1 前最先看。
2. **專案狀態列**：每個參與專案一張小卡，只標「活躍 / 維護 + 有無待確認」，中性陳述，不評量。
3. **本週重點**：跨專案合併，每條前面用小標籤標來源專案。
4. **成長面向**：跨專案合併。
5. **待確認彙整**：所有專案的待確認收在一起，**按專案分組**呈現（見 2.5）。

### 2.4 各專案 Tab（單一專案完整內容）

該專案下這個人的完整報告：本週重點、成長面向、待確認項目。
頂部以 pill 標示本專案的概況（例「game-backend · 6 MR」）。

### 2.5 待確認項目呈現方式（已定稿）

採 **按專案分組**：每個專案一個小標頭（資料夾 icon + 專案名），下面列該專案的待確認 checkbox 項目。
理由：1on1 談話自然以專案為單位切換脈絡，分組對齊談話節奏；視覺結構也與「本週重點 / 成長面向」一致。

待確認區塊使用 warning 色系，與一般內容區隔。每項為一個 checkbox（`☐` 樣式），措辭為開放式問句，供管理者當面詢問。

### 2.6 「趨勢」Tab（長期觀察，純質性）

**不含任何數量型圖表。** 三個區塊。**資料來源已定：直接讀既有 1on1 檔案**，不在 SQLite 另建趨勢表。

| 區塊 | 主要資料來源 | 說明 |
|------|-------------|------|
| 長期觀察 | `index.md` + 歷史 `YYYY-MM.md` | AI 跨月綜合敘述；可即時算或快取 `_cache/long_term.md`（見 §8 #4） |
| 成長軌跡 | 歷史 `YYYY-MM.md`（跨季綜合） | 時間線標記能力 / 職責關鍵轉變；最新節點高亮 |
| 歷史待確認追蹤 | `_notes.md` + 各月報告中的待確認區段 | 解析過月疑慮與閉環狀態 |

1. **長期觀察**：講「此人如何面對問題」「技術深度如何累積」「成長方向」，而非產量。以引文式左邊框呈現。
2. **成長軌跡**：以季為單位的時間線（例：功能開發 → 效能議題 → 主導重構）。
3. **歷史待確認追蹤**：每項標月份 + 狀態：
   - ✓ 已改善（success 色）
   - ○ 待確認 / 待本次 1on1 跟進（warning 色）

   此區塊價值最高：避免 1on1 反覆講同樣問題卻無人追蹤，把疑慮變成可閉環的迴圈。

> 本週「待確認」的即時狀態由 SQLite **`pending_items`** 驅動（schema §3.2）；趨勢 Tab 的**歷史與長期敘述**以檔案為準。

### 2.7 報告產出格式（skill 端）

每次執行，reviewer skill 對每位工程師產出（路徑見 §9.0）：

- `report.md`：完整報告，供深讀 / 存檔。
- `summary.md`（傾向，待決 #3）：精簡報告，供本週 Tab；含 frontmatter + `## 本週重點` / `## 成長面向` / `## 待確認`（格式見 schema §7）。

**分工原則**：AI 只負責「篩重點」，排版交給前端；Markdown 由**後端**以 `pulldown-cmark` 渲染後提供 API，不需 AI 產 HTML。

> 待決 #3：若改採 `report.md` 內標記區段，則取消獨立 `summary.md`，後端解析規則需同步調整。

## 3. 控制台首頁

登入後的落地頁，快速總覽。

- **頂部**：上次執行時間 + 耗時；「立即執行」（全部）按鈕。
- **統計卡**（5 張）：專案數、工程師數、未讀報告數、待確認總數、MR 草稿數（`mr_reviews.status='draft'`）。
  - 註：這些是「狀態計數」（幾份未讀、幾項待確認），非「產量指標」，與 §0 非目標不衝突。
- **最近報告**：列出最新一批報告，標未讀（藍點）/ 已讀、待確認數。點擊跳到該報告。
- **排程卡**：週期（例「每週一 09:00」）、下次執行時間、排程器運行狀態。

## 4. 專案設定

### 4.1 左欄：專案清單（兼執行入口）

每列同時是設定入口與執行入口：

- Hover 顯示「▶ 執行」：單獨執行該專案（見 §6）。
- 正在執行：轉圈 icon + 計時。
- 閒置：顯示上次執行日期。
- 頂部「全部」：一次執行所有專案。「＋」：新增專案。

### 4.2 右側：專案設定表單

欄位對應 **`projects` 表**（schema §2.3），UI 標籤如下：

1. **專案名稱** → `name`：顯示用、報告閱讀器 Tab 名；亦作 `reports/<name>/` 目錄名。
2. **`repo_path`**：伺服器上 git working copy 的絕對路徑。慣例：`$REVIEWER_DATA_DIR/repos/<slug>/`（例 `/data/reviewer/repos/game-backend`）。skill 子行程 `cwd` 設為此目錄；`glab` 於此執行。
3. **`git_remote_url`**（選填）：Git remote URL；待決 #7 是否由後端自動 `clone` / `pull`。
   - 儲存後偵測：確認是 git repo（`is_git_repo`）、顯示 `default_branch`。
4. **參與工程師**（唯讀）：由 `person_identities` 比對得出，對應 **`participation` 表**。

> 來源類型（GitLab vs 本地）已移除。所有專案一律伺服器上的 git working copy；MR 資料靠伺服器已登入的 `glab` CLI 取得，web 介面不管理 GitLab token。

## 5. 人員設定

歸戶以人為中心，集中管理，避免每個專案重複設定。

### 5.1 左欄：人員清單
列出所有工程師，標參與專案數。「＋」新增。

### 5.2 右側：人員表單

1. **顯示名**：1on1 與報告中顯示的名字（例「Alice」）。
2. **Git Author Identity（可綁多個）**：commit / MR 歸戶的依據。
   - 一個人可綁多筆 identity：公司 email、個人 email、`glab` username 等。
   - 每筆可加來源標籤（公司 / 個人 / glab user）以利辨識。
   - 比對邏輯：commit 或 MR 的 author 命中任一筆 identity，即歸到此人。
3. **參與專案**（唯讀）：由 identity 自動比對「哪些專案的 commit 含此人」反推得出。

### 5.3 設計理由
- 新人入職：只在人員頁設定一次 identity，即在所有專案自動歸戶。
- 換 email / 多機器 git config：補一筆 identity，全專案生效。
- 專案增減：完全不需碰人員設定。
- 維護成本由「專案 × 人員」降為「專案 + 人員」。

> 實作注意：同一人可能有多個 email，或 `glab` 的 MR author 與 commit author email 不一致。identity 設計為「一個顯示名綁多個 author identity」即為此。需處理「同一 identity 不可同時綁到兩個人」的衝突驗證。

## 6. 執行模型

### 6.0 skill／後端邊界原則（貫穿全案）

- **skill（`claude -p` 子行程跑的）只碰檔案**：讀 repo / MR，寫 `summary.md`、MR 草稿、觀察片段，以及對齊既有結構的 `index.md` / `YYYY-MM.md`。skill **不查 SQLite**（headless 下查庫不自然）。
- **後端（Rust 常駐服務）獨佔 SQLite**（設定、執行狀態、收件匣、已讀、本週 pending），並**讀 skill 產的檔案**來渲染 API 與解析入庫。
- **趨勢頁長期資料**：後端讀檔提供 API，**不**在 SQLite 累積 trend 快照（見 §2.6）。
- **兩軌之間、skill 與後端之間的資料交換一律走檔案**，skill 不共享 DB。

**執行機制**：後端以子行程呼叫 `claude -p "<觸發 skill 的 prompt>"`，`cwd` = `projects.repo_path`。`claude` / `glab` 須在伺服器 PATH 或後端設定指定。逾時對應 `schedule_config.per_project_timeout_sec`（§7.1）。

### 6.1 兩條軌道

| | 軌道 1：1on1 週報 | 軌道 2：MR review |
|---|---|---|
| 節奏 | 每週一次（`schedule_config`） | 輪詢（`mr_poll_interval_min`，預設 60 分鐘） |
| 對象 | 專案 repo 的 commit | 伺服器 `glab` 抓待 review 的 MR |
| skill | 新做 `reviewer-batch`（唯讀） | headless 改造 `cto:scan-mrs`（唯讀，見 §6.4） |
| 產出 | 結構化 `summary.md` → 閱讀器 | review 草稿 → 收件匣；觀察片段 → 打通週報 |
| 副作用 | 無 | 唯讀；**發佈需人工按鈕** |

### 6.2 軌道 1：觸發與狀態

**兩種觸發**：
- **全部執行**：跑所有專案（排程自動觸發的也是這個）。入口在控制台首頁、專案清單頂部。
- **單一專案立即執行**：入口在專案清單的每一列（hover）。單專案跑完即更新該專案報告，不等待其他專案。

**各專案執行狀態**：分析中（轉圈 + 計時 + 處理中的人）/ 完成（耗時，可重跑）/ 佇列中（可插隊）/ 閒置（上次執行日期）。

### 6.3 並發模型（實作考量）
單專案獨立執行與全部執行可能同時發生，背後需要 **任務佇列 + 執行器**，而非單純 for loop：

- **去重鎖**：同一專案不可同時被排入兩次（全部執行進行中又手動點同一專案）。MR 輪詢與週報也共用此鎖，避免同專案同時被兩軌 clone/checkout 打架。
- **並發上限**：同時執行多個 Claude Code 很吃資源，建議 worker pool（例：同時最多 2 個），其餘排佇列。兩軌共用同一 worker pool。
- Rust 實作方向：async task + semaphore 控制並發；後端常駐於單一（或少量）雲端 worker，並發上限依主機資源設定。

### 6.4 軌道 2：MR review（headless 改造 scan-mrs）

現有 `cto:scan-mrs` 是**互動式**（每步問確認）、**有副作用**（`glab mr note` 發佈、`glab mr merge`）、以「指派給我的 MR」為視角。後端無法直接無人值守跑它。做法：**拆出一個 headless 唯讀模式**，複用其高價值邏輯，砍掉互動與副作用。

**保留（高價值）**：
- **分輪判斷**（第一輪 / 第二輪）與**「無新動靜就跳過」的去重**——靠 GitLab notes 中 `By: AI Agent` 判斷。**此去重必須保留**，否則高頻輪詢每次重跑，token 燒穿。
- 事實查核、同類模組對照、六大面向審查。

**移除 / 改寫**：
- 移除所有互動確認（Step 3 / 4f / 4h / 4i）。
- 移除 `glab mr note` / `glab mr merge` 副作用——skill **只產草稿檔**，不寫 GitLab。
- 輸出改為落檔（見下），供後端解析入庫，不再走「詢問是否寫入 1on1」流程。

**產出（落檔）**：
1. **MR 草稿** → 後端解析進 `mr_reviews`（收件匣）。管理者在 web 收件匣看過可**編輯 / 發佈（`glab mr note`）/ 忽略**；發佈才碰 GitLab。
2. **觀察片段** → `$REVIEWER_DATA_DIR/reports/<name>/<person>/_pending/`（schema §0），供週報消費（§6.5）。

### 6.5 兩軌打通（MR 觀察 → 週報）

軌道 2 每次跑，除了草稿，另抽「工程師觀察片段」落到 `_pending/`。軌道 1 週報 `reviewer-batch` 跑時：掃 commit code review **＋** 讀本週累積的 MR 觀察片段，合併成 `summary.md`，消費後將片段歸檔。

**兩條契約（推薦值，待最終確認）**：
- **觀察何時算進週報**：**發佈後才算**——只有管理者在收件匣發佈的 MR review，其觀察片段才進週報。等於人工過濾掉 AI 誤判。（待確認；替代：產生時就算。）
- **去重**：週報彙整時，**同一 MR 只取最新一輪**觀察，避免同一件事重複塞進「本週重點」。（待確認。）

## 7. 排程

### 7.1 設定（對應 **`schedule_config` 表**，schema §2.5）

- `enabled`：自動執行開關。
- `cadence`：每日 / 每週 / 每兩週。
- `weekday`：星期（每週 / 每兩週時；0=週一）。
- `run_time`：時間（例 `09:00`）。
- `mr_poll_interval_min`：MR 輪詢間隔（分鐘，預設 60）。
- `per_project_timeout_sec`：單專案逾時（預設 600）。
- `max_concurrency`：worker pool 並發上限（預設 2）。

### 7.2 狀態與操作
- 下次執行時間 + 倒數。
- 排程器運行狀態指示。
- 立即執行一次（全部）。
- **單專案逾時**：超過 `per_project_timeout_sec` 即 kill 子行程、標 `run_projects.state='skipped_timeout'`。
- 最近執行紀錄：成功 / 有逾時跳過，含日期、專案數、耗時。

### 7.3 排程實作方式與「漏跑補償」（已定：後端內建）

- 採 **後端內建常駐排程**（`tokio-cron-scheduler` 或同等方案），隨 Rust 服務啟動，無需使用者本機常駐或系統匣。
- **雲端優勢**：排程與 MR 輪詢只要服務在跑即生效；與桌面 app「必須開著才觸發」的限制不同。
- **漏跑補償（建議納入）**：服務**重啟時**檢查「上次應執行時間已過、但無對應執行紀錄」，若是則在控制台提示「上週排程未跑，是否補跑」。主要覆蓋部署重啟、主機當機等短暫中斷，而非使用者忘記開 app。

## 9. 資料來源與儲存佈局

> 欄位與表結構詳見 **schema.md**；本節定義 spec 與 schema 共用的路徑與檔名約定。

### 9.0 伺服器目錄（`REVIEWER_DATA_DIR`）

```
$REVIEWER_DATA_DIR/                   # 例 /data/reviewer
├── reviewer.db                       # SQLite
├── repos/                            # projects.repo_path 指向此下
│   └── <slug>/
└── reports/
    └── <name>/                       # 通常 = projects.name
        └── <person>/                 # 通常 = people.display_name
            ├── index.md
            ├── YYYY-MM.md
            ├── _notes.md
            ├── _pending/             # MR 觀察片段（軌道 2）
            ├── _cache/               # 選填：long_term.md 等 AI 快取
            └── <YYYY-MM-DD>/
                ├── report.md
                └── summary.md
```

### 9.1 1on1 檔案（趨勢 + skill 長期產出）

| 檔案 | 用途 |
|------|------|
| `index.md` | 累積型長期描述 → 趨勢「長期觀察」 |
| `YYYY-MM.md` | 月度記錄 → 「成長軌跡」、跨月綜合素材 |
| `_notes.md` | 備註 / 追蹤 → 趨勢「歷史待確認」 |
| `report.md` / `summary.md` | 單次執行產物 → 本週 Tab；路徑冗餘存 **`reports` 表** |

趨勢 API **讀檔**，不查 trend 相關 SQLite 表。

### 9.2 執行時資料來源

- **commit / diff**：`projects.repo_path` 下的 git working copy。
- **MR**：伺服器 `glab` CLI（`cwd` = `repo_path`）。
- **本週報告 / 收件匣**：skill 落檔 → 後端解析入 **`reports`**、**`mr_reviews`**、**`pending_items`**。

### 9.3 MVP 設定檔（`projects.yaml`）

MVP 手編 yaml，欄位對應 `projects` 表，匯入或啟動時載入：

```yaml
projects:
  - name: game-backend
    repo_path: /data/reviewer/repos/game-backend
    git_remote_url: git@gitlab.example.com:team/game-backend.git  # 選填
```

## 10. MVP 範圍建議（漸進式）

| 階段 | 範圍 |
|------|------|
| **MVP** | Rust 後端 + 瀏覽器前端；報告閱讀器（本週）+ web 通知 + `projects.yaml`（`name` / `repo_path` / `git_remote_url`）+ `REVIEWER_DATA_DIR` + 後端排程 + 全部執行 |
| **v2** | 專案 / 人員設定 UI（取代手編 yaml）+ identity 歸戶 + 單專案立即執行 + MR 收件匣 |
| **v3** | 趨勢頁（讀檔：長期觀察 / 成長軌跡 / 歷史待確認）+ 排程 UI + 漏跑補償 + 控制台統計 + 多使用者認證 |

理由：設定 UI 是工程量膨脹最快的地方。先用手編 yaml 驗證「核心流程真的有用、會每週看」，再往上加 UI，避免卡在設定頁而核心價值未驗證。

## 附錄：設計原則速記

- **以人為中心**：1on1 面對的是人，不是專案。左欄列人，歸戶集中於人員設定。
- **質重於量**：看方向與成長，不看產量。無數量圖表，不點落差。
- **中性陳述**：活躍 / 維護是事實描述，非評價。
- **閉環**：待確認項目可跨月追蹤是否改善。
- **分工**：AI 篩重點，前端管排版。
- **漸進**：先驗證核心價值，再擴張設定 UI。
