# 工作流與人員成長 — 功能路線圖

> 狀態：**規劃建議**（尚未開 OpenSpec change）
> 用途：對照現有規格與實作缺口，提出能強化工作流與人員成長的功能建議。
> 原則：以人為中心、質重於量、閉環追蹤、人機協作。
> 備註：部分項目已在 `spec.md` 規劃但尚未落地，優先補齊會比加新功能更有感。
> 日期：2026-07-10

---

## 一、先補規格缺口（投資報酬率最高）

這些功能在設計裡已有，補上後整體工作流才算「閉環」。

### 1. 待確認項目閉環 UI + API

目前 `pending_items` 會從週報寫入、在左欄與控制台顯示數量，但**沒有**把項目標成 `resolved` 的 API 與 checkbox 操作。

建議：
- `PATCH /api/pending-items/:id`（`resolved` + 選填 `resolution_note`）
- 報告閱讀器「待確認」區塊可勾選閉環
- 閉環時同步寫回 `_notes.md`（spec 已提到），讓趨勢 Tab「歷史待確認」能追蹤

**價值**：把 1on1 從「每週看報告」升級為「跨月追蹤成長議題是否改善」——這是 spec 強調的核心閉環。

### 2. 完整人員設定頁

現有「未歸戶」面板能事後綁定，但缺少獨立人員 CRUD、identity 管理、參與專案唯讀檢視。

**價值**：新人 onboarding 一次設定、全專案生效，降低歸戶漏網。

### 3. 完整排程設定 UI + 漏跑補償

MR 輪詢間隔可改，但週報 cadence / weekday / timeout / concurrency 仍偏硬編碼；spec §7.3 的「重啟後提示補跑」也未做。

**價值**：雲端服務重啟、部署後不會默默漏掉一週觀察。

### 4. 執行紀錄與可觀測性

控制台有「上次執行」，但缺少完整的 runs 歷史、逾時/失敗原因、MR triage skip 明細（`inbox_draft`、`no_new_activity` 等）。

**價值**：除錯與信任——管理者要知道「為什麼這週沒報告 / 為什麼 MR 沒掃」。

## 二、強化主管工作流（1on1 前後）

### 5. 1on1 準備模式（Prep View）

在報告閱讀器加一個「準備 1on1」視圖，自動彙整：
- 此人所有 open 待確認（跨專案分組）
- 本週一句話摘要（`one_line`）
- 趨勢 Tab 最近一條成長軌跡節點
- 相關 MR 草稿（若此人近期有 MR 被 review）

可一鍵匯出 Markdown / 複製到剪貼簿，方便開會前 5 分鐘掃一眼。

**價值**：把分散在四個 Tab 的資訊收成「開會議程」，直接服務 1on1 場景。

### 6. 1on1 會後紀錄（Session Notes）

每次 1on1 後，主管可記錄：
- 哪些待確認已當面釐清
- 約定的下一步（非 KPI，而是質性方向，如「下季多帶 junior review」）
- 寫入 `reports/_people/{name}/_sessions/YYYY-MM-DD.md` 或 SQLite `session_notes`

下週 AI 跑週報時，manifest 帶入最近 session notes 作為脈絡。

**價值**：人類判斷與 AI 觀察銜接，避免每週從零開始分析。

### 7. 新報告通知

MVP 規格有提 web 通知，目前主要靠 polling。

建議輕量方案：
- 瀏覽器 Notification API（需使用者授權）
- 或 Slack / Teams webhook：「本週 12 份報告已就緒，3 份未讀、5 項待確認」

**價值**：讓系統從「要記得去開」變成「主動提醒你該看」。

### 8. MR 收件匣工作流強化

已有編輯 / 發佈 / 追問，可再加：

| 功能 | 說明 |
|------|------|
| **草稿 diff** | 發佈前對比 AI 原版 vs 主管修改，累積「常改什麼」 |
| **Force rescan** | 對 `ignored` 或已發佈但 MR 有新 commit 的項目，手動強制重掃 |
| **批次操作** | 多選 ignore / 標記「稍後處理」 |
| **發佈前檢查清單** | 確認是否含建設性回饋、是否涉及敏感議題 |

**價值**：降低 MR review 摩擦，讓軌道 2 真正進入日常節奏。

## 三、深化人員成長（符合「質重於量」原則）

### 9. 成長主題標記（Qualitative Themes）

不計 commit 數，而是讓 AI（或主管手動）為每人標記**質性主題**，例如：
- 「資料庫效能意識」
- 「跨模組重構主導」
- 「review 回饋品質提升」

存在 `YYYY-MM.md` 或輕量 `growth_themes` 表，趨勢 Tab 用時間線呈現主題演進（文字時間軸，不是圖表）。

**價值**：把零散的週報觀察收成「這個人在哪條成長軌道上」。

### 10. 待確認老化提醒

open 超過 N 週（如 4 週）的待確認，在左欄與控制台標「逾期未追」。

**價值**：避免同一問題反覆出現在週報卻從未在 1on1 處理。

### 11. 長期觀察快取策略落地

spec §8 #4 待決：長期觀察是每週重算、每月重算，還是請求時即時算。

建議：**每月重算 + 請求時 stale 則背景刷新**，寫入 `_cache/long_term.md`。

**價值**：趨勢 Tab 回應更快，且跨月敘述更連貫。

### 12. 跨專案「總覽 Tab」補齊

spec §2.3 定義了總覽 Tab（一句話摘要、專案狀態列、跨專案重點合併），需確認前端是否完整實作；若只有各專案 Tab，應補上。

**價值**：跨專案工程師的 1on1 不必逐 Tab 切換。

## 四、營運與信任（上線前必要）

### 13. 多使用者認證 + 操作歸屬

spec 明確延後，但多位主管共用時需要：
- 誰發佈了哪則 MR note
- 誰標記了待確認閉環
- 誰編輯了草稿

可先 MVP：`shared secret` + 簡單 `manager_name` 選擇；正式版再接 SSO。

### 14. Identity 智慧建議

未歸戶佇列出現 `alice@personal.com` 時，依 git name / email 相似度建議「可能同一人」。

**價值**：降低歸戶維護成本，減少「跑了 review 卻沒週報」的情況。

### 15. 健康檢查儀表板

擴充 `/health` 或控制台：
- 各專案 `is_git_repo`、最後 fetch 時間
- agent CLI 是否可用
- `glab` auth 是否有效
- 磁碟空間（已有 `REVIEWER_MIN_FREE_BYTES`）

**價值**：問題在主管發現「沒報告」之前就暴露。

## 五、建議路線圖

```
Phase A（閉環補齊，1–2 週）
  待確認閉環 UI
  → 人員設定頁
  → 執行紀錄 / skip 明細
  → MR E2E 驗證收尾

Phase B（1on1 工作流，2–3 週）
  1on1 準備模式
  → 會後 session notes
  → 新報告通知
  → 待確認老化提醒

Phase C（成長深化，持續）
  成長主題時間軸
  → 長期觀察快取
  → 總覽 Tab 補齊
  → MR 草稿 diff / force rescan

Phase D（上線準備）
  認證 + 操作歸屬
  → 健康檢查儀表板
  → identity 智慧建議
```

## 六、刻意不建議的方向

對齊 spec 非目標，以下不建議做：

| 方向 | 原因 |
|------|------|
| Commit/MR 數量圖表 | 與「質重於量」衝突 |
| 自動發佈 MR / 自動 merge | 人機協作底線 |
| Telegram 雙向問答 | spec 已排除，成本高、效益低 |
| 跨人排名 / 績效評分 | 偏考核工具，偏離 1on1 成長輔助定位 |
| 全自動補跑所有漏掉的週報 | 可能產出過時報告；改為「提示 + 手動確認」較好 |

---

## 總結

若只能選 **3 個** 最能強化工作流與成長的功能：

1. **待確認閉環** — 讓「觀察 → 1on1 追蹤 → 跨月驗證改善」真正跑通
2. **1on1 準備模式 + 會後紀錄** — 把 AI 產出嵌入主管真實管理節奏
3. **成長主題時間軸** — 在不做數量指標的前提下，給出「這個人在往哪裡成長」的長期敘事

這三項都符合現有架構（檔案為主、SQLite 管狀態、雙軌打通），工程量可控，且直接服務「人員成長」這個產品核心。

若要往某個方向開 OpenSpec change proposal，可以指定優先項，再起草 `proposal.md` 初稿。
