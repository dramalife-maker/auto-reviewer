## Context

MR 素材由 worker 在派工 agent 前預先產生（log / stat / diff 三個檔），其中 diff 有 48 KiB 的位元組上限，超過即截斷並附上截斷標記。實務上鎖檔、產生碼、vendor 目錄這類雜訊檔動輒數千行，會把額度吃光，使真正需要 review 的程式碼落在截斷點之後。

現有的 per-project 設定（例如 MR 略過標籤）由 projects.yaml 在啟動時 upsert 進 projects 表，每次啟動都以 YAML 覆寫 DB 欄位，因此不適合承載「UI 可編輯、即時生效」的設定。相對地，排程設定採單列表加 API 的形式，已具備前端可編輯且不受 YAML 干擾的先例。

週報 agent 與 MR agent 皆持有自由 shell，會自行執行 git 指令探索變更，後端無法從外部強制約束其讀取範圍。

## Goals / Non-Goals

**Goals:**

- 提供一份全域的忽略清單，內容為 git pathspec 字串，儲存於資料庫。
- 前端可編輯並即時儲存，變更於下一場 run 生效，不需重啟服務。
- MR 素材的 diff 內容確實排除清單所列檔案，且被忽略的檔案仍在 stat 中可見。
- 單一錯誤的 glob 不得導致 MR 批次靜默中止。

**Non-Goals:**

- 不提供 per-project 粒度的忽略清單。雜訊檔在各 repo 高度同構，per-project 需要改動 YAML 同步的覆寫語意，複雜度不成比例。
- 不對 agent 自行執行的 git 指令做強制過濾。agent 具備自由 shell，任何後端過濾都可被繞過，此處僅以 manifest 與 WORKFLOW 指示提供軟約束。
- 不為週報預先產生過濾後的 git 素材。週報是 per-person 的探索式閱讀，沒有固定素材形狀可預算，且週報不受 diff 位元組上限影響，缺乏此功能的核心動機。
- 不提供預設忽略項目。

## Decisions

### 忽略清單採全域單列設定表而非 per-project 欄位

新增 review_settings 表，固定單列 id=1，欄位 ignore_globs 以 JSON 陣列字串儲存，與既有 MR 略過標籤欄位的存法一致。

替代方案是加在 projects 表上做 per-project 設定，但 projects 表由 projects.yaml 在啟動時全欄覆寫，UI 寫入的值會在下次啟動被蓋掉；要支援 UI 編輯就得為個別欄位引入「YAML 僅供初始值」的例外語意，使同步規則不再一致。另一個替代是擴充既有的排程設定表，可省下一支 migration，但該表語意為排程，塞入 review 過濾規則會造成長期誤導。

### 忽略只作用於 diff 內容，stat 與 log 保留完整

產生素材時僅在輸出 change.diff 的 git diff 指令附加排除用 pathspec；產生 change_stat.txt 的 git diff --stat 與產生 change_log.txt 的 git log 不附加。

如此被忽略的檔案仍會以檔名與增刪行數出現在 stat，agent 知道它被改過、規模多大，必要時可自行針對該路徑取 diff。全部排除的方案會讓 agent 完全看不見這些檔案，一旦使用者填入過寬的規則（例如整個目錄），review 可能在無自覺的情況下漏判。本功能的核心價值在於保護 diff 的位元組額度，僅排除內容即已達成。

### pathspec magic 前綴由後端組裝，拒絕使用者自帶前綴

儲存的是原始字串，送入 git 前才由後端包成排除語法。驗證規則拒絕以冒號開頭的輸入，避免組出雙重 magic 前綴而使 git 指令失敗。不附加嚴格 glob magic，讓萬用字元可跨目錄層級匹配，符合使用者對副檔名規則的直覺預期。

其餘正規化：去除前後空白、丟棄空字串、去除重複、單條長度上限 200 字元、清單總數上限 100 條。驗證集中在後端 API 層，前端僅做即時提示，不重複實作規則。

### 帶 pathspec 的 diff 失敗時降級重跑

若附加 pathspec 的 git diff 指令回傳失敗，記錄警告後改以不帶 pathspec 的形式重跑一次；若仍失敗才回報原有的錯誤。現行流程中素材產生失敗會使該 MR 被跳過，若無降級，一條打錯的規則會讓所有 MR 連續靜默跳過，只留下日誌痕跡。

### 設定以全量覆蓋方式更新

提供讀取與更新兩支端點，更新為全量覆蓋語意：請求帶入完整清單，伺服器正規化後取代原值並回傳結果。清單型設定用全量取代最單純，不需要為新增與刪除設計差異操作。

### 設定於每場 MR 掃描開始時讀取一次

worker 在進入 MR 迴圈前讀取一次清單並沿用於該場 run 的所有 MR，避免每個 MR 重複查詢。因為讀取發生在 run 開始，UI 改動於下一場 run 生效，不需重啟服務。

### 週報與 MR agent 透過 manifest 取得清單並在 WORKFLOW 中約束

兩種 manifest 皆新增 ignore_globs 欄位，並在週報與 MR 兩份 WORKFLOW 文件加入指示，要求 agent 自行執行的 git 指令一律附帶排除用 pathspec。此為軟約束，效力取決於 agent 遵循程度。

### 前端以獨立卡片與獨立儲存動作呈現

在 Dashboard 排程卡片下方新增獨立卡片，內含每行一條的文字框與自己的儲存按鈕。因端點與排程不同，無法併入排程的儲存流程。文字框以 placeholder 提示常見寫法，預設值為空清單。儲存後提示訊息需說明變更於下一場 run 生效。

## Implementation Contract

**行為**

- 操作者在 Dashboard 的 Review 過濾卡片輸入每行一條的忽略規則並儲存後，下一場 MR 掃描產生的 change.diff 不再包含符合規則之檔案的差異內容。
- 同一份素材的 change_stat.txt 仍列出這些檔案的檔名與增刪統計，change_log.txt 內容不受影響。
- 清單為空時，素材產生行為與本變更前完全相同。

**介面與資料形狀**

- 資料表 review_settings：主鍵 id 固定為 1，欄位 ignore_globs 為 NOT NULL 的文字欄位，內容是 JSON 字串陣列，預設值為空陣列字面值。migration 需寫入對應的 schema 版本紀錄，並插入 id=1 的初始列。
- GET /api/review-settings 回傳 JSON 物件，含 ignore_globs 字串陣列欄位。
- PUT /api/review-settings 接受同形狀的 JSON 物件，正規化後全量覆蓋，並以正規化後的結果作為回應內容。
- 素材產生函式新增一個字串切片參數承載忽略清單；既有呼叫端傳入從資料庫讀出的清單，測試替身路徑傳入空清單。
- 兩種 manifest 結構新增 ignore_globs 字串陣列欄位，清單為空時序列化可略過。

**失敗模式**

- PUT 收到不合法項目時回傳 400，訊息需指出違規原因（冒號開頭、超長、超過筆數上限）。空白與重複項目不視為錯誤，正規化時直接移除。
- 附加 pathspec 的 diff 指令失敗時不向上拋錯，改為記錄警告並以不帶 pathspec 重跑；重跑仍失敗才回報既有的 git 錯誤，由呼叫端沿用現行的跳過處理。
- stat 與 log 指令的失敗處理維持現狀，不因本變更改變。

**驗收條件**

- 後端單元測試：在具備符合忽略規則之檔案的測試倉庫上產生素材，斷言 diff 檔不含該檔案的差異內容，且 stat 檔仍含該檔案名稱。
- 後端單元測試：傳入會使 git 失敗的規則時，素材仍成功產出且 diff 內容等同不帶規則的結果。
- 後端單元測試：驗證正規化與驗證規則，涵蓋去空白、去重、冒號開頭被拒、超長被拒、超過筆數上限被拒。
- 後端整合測試：對更新端點送出清單後，讀取端點回傳正規化後的相同內容。
- 前端測試：於卡片輸入內容並觸發儲存，斷言以正規化前的原始輸入呼叫更新端點，且成功後顯示提示訊息。
- 手動驗證：清單留空時，一場 MR 掃描產生的素材與本變更前一致。

**範圍邊界**

- 在範圍內：資料表與 migration、兩支 API 端點與其驗證、MR 素材產生的 diff 過濾與降級、兩種 manifest 的欄位、兩份 WORKFLOW 的指示段落、Dashboard 的編輯卡片、上述測試。
- 在範圍外：per-project 忽略設定、週報素材預產、agent 端的強制過濾、diff 位元組上限本身的調整、projects.yaml 結構變更。

## Risks / Trade-offs

- [使用者填入過寬的規則，導致重要變更的內容未被 review] → 保留 stat 與 log 使被忽略的檔案仍然可見；預設為空清單，不主動製造隱形行為；UI 提示說明規則僅排除 diff 內容。
- [agent 不遵守 WORKFLOW 的軟約束，自行取得完整 diff] → 接受此限制。後端素材已完成主要的額度保護，agent 自行讀取屬額外成本而非正確性問題。
- [降級機制掩蓋了設定錯誤，使用者不知道規則無效] → 降級時記錄警告層級日誌並標明失敗的規則來源，使問題可從日誌追查。
- [全域設定無法滿足未來的 per-project 需求] → 新表為獨立單列設定表，日後若確有需求，可再引入 per-project 覆寫而不需重構既有欄位。
